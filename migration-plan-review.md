Findings

Critical: the auth and permissions plan removes the current local-install identity model, which will break unauthenticated/offline data ownership and the existing merge-on-login/logout behavior. The plan assumes Appwrite user auth can replace stored local user IDs and per-document access control directly via authenticated users in migration-plan.md:13, migration-plan-in_detail.md:66, but the app currently uses a device/local user ID as the active identity when logged out, promotes that data into an account on login, and snapshots account data back to the local identity on logout in AuthContext.tsx:204, AuthContext.tsx:258, AuthContext.tsx:512, AuthContext.tsx:544, AuthContext.tsx:242, and AuthContext.tsx:248. You need an explicit replacement for “local install identity” before this migration is viable.

High: the plan treats notes and reminders as separable feature migrations, but they are the same underlying data model today. The phased split in migration-plan.md:25 and migration-plan.md:33, plus the “feature-flag per feature” idea in migration-plan.md:119, conflicts with the current schema where reminder fields live on the notes table in schema.ts:25, schema.ts:36, schema.ts:50, and with reminder queries/mutations that operate on notes in reminders.ts:46. If you cut over notes and reminders independently, you create split ownership of the same documents and high risk of divergence.

High: the client-side CRUD approach drops important server-side consistency guarantees. The plan explicitly moves notes/reminders/subscriptions CRUD to the client in migration-plan-in_detail.md:84, migration-plan-in_detail.md:123, migration-plan-in_detail.md:142, but current writes are not “just CRUD”: syncNotes batches changes, emits change events, and returns canonical state in notes.ts:14, notes.ts:67, while reminder mutations write the note, append an audit event, and enqueue push work from the backend in reminders.ts:67, reminders.ts:117, reminders.ts:127, reminders.ts:230, reminders.ts:312. Mobile sync also assumes one batched backend call that returns authoritative server versions in syncQueueProcessor.ts:205. Pushing these flows to direct client writes creates partial-failure windows unless you redesign them around server functions.

High: the provider/hook swap underestimates how much of the app is coupled to Convex outside React hooks. The plan focuses on replacing ConvexProvider, useQuery, and useMutation in migration-plan.md:73, migration-plan-in_detail.md:251, but there are imperative and background clients in auth, sync, voice, push-token registration, headless reminder actions, trash operations, and fetch helpers: AuthContext.tsx:102, syncQueueProcessor.ts:307, registerDeviceToken.ts:175, fetchNotes.ts:30, fetchReminder.ts:31, headless.ts:65, aiIntentClient.ts:87, AuthContext.tsx:64. This migration needs an adapter-inventory phase before implementation, otherwise the cutover will be incomplete.

Medium: the cron/reminder plan drops compatibility behavior that is still required during an incremental migration. The proposed due-reminder logic only mentions nextTriggerAt and snoozedUntil in migration-plan.md:52 and migration-plan-in_detail.md:183, but the live cron still checks legacy triggerAt and derives a stable eventId from snoozedUntil ?? nextTriggerAt ?? triggerAt in reminderTriggers.ts:72 and reminderTriggers.ts:227. During dual-write or staggered rollout, dropping that fallback will miss old reminders or change dedup semantics for delivered notifications.

Medium: the test migration scope is incomplete. The plan only calls out contract tests and one integration test in migration-plan-in_detail.md:283, but mobile unit tests also mock Convex browser/API types in aiIntentClient.test.ts:2 and fetchNotes.userIdMapping.test.ts:5, and those will need migration too.

Assumptions
I reviewed the plan against the current repo shape and runtime behavior. I did not validate Appwrite product limits/API details from live docs here, so there may be additional platform-specific constraints on top of the repo-specific issues above.

Change Summary
The plan is directionally plausible, but it is not implementation-safe yet. The biggest gap is that it assumes “Convex backend” maps cleanly to “Appwrite auth + client CRUD + a few functions,” while this codebase currently relies on a local-install identity model, server-mediated merge flows, and note/reminder logic sharing the same records.

1. Add a pre-phase that defines the replacement for local/device identity and logout/login merge semantics.
2. Merge “notes” and “reminders” into one migration track, because they are one data model.
3. Replace the current “client-side CRUD” approach for write-heavy flows with server-owned endpoints/functions, then do a second pass on the plan.
