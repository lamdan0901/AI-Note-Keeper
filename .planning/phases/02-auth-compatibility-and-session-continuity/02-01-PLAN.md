<threat_model>

## Trust Boundaries

| Boundary      | Description                                                                         |
| ------------- | ----------------------------------------------------------------------------------- |
| client->api   | Untrusted credential and token inputs cross into auth crypto and persistence logic. |
| api->postgres | Token lifecycle state is enforced by DB reads/writes and uniqueness constraints.    |

## STRIDE Threat Register

| Threat ID | Category | Component                | Disposition | Mitigation Plan                                                                                                                        |
| --------- | -------- | ------------------------ | ----------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| T-02-01   | S        | token verification       | mitigate    | Verify signature, issuer, audience, and expiry for every token operation using jose-based verifier wrapper.                            |
| T-02-02   | T        | refresh token storage    | mitigate    | Store hashed refresh tokens only, enforce unique token_hash, and never persist raw token values.                                       |
| T-02-03   | I        | password migration path  | mitigate    | Accept legacy salt:sha256 verification only for login compatibility and immediately emit argon2id upgrade signal on success.           |
| T-02-04   | E        | session revocation logic | mitigate    | Revoke only active token record/lineage per D-06 and test cross-session isolation to prevent privilege-impacting over-revocation bugs. |

</threat_model>
