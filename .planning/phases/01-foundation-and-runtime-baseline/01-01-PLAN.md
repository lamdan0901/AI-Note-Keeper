<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client->api | Untrusted HTTP payloads cross into validation and error handling layers. |
| api->postgres | Service trust depends on DB availability and migration state truthfulness. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | T | request validation boundary | mitigate | Apply schema-first validation middleware at route entry and return `validation` contract with sanitized details only. |
| T-01-02 | I | error middleware | mitigate | Never serialize stack traces/internal exception content into `details`; only allow curated fields for client-correctable errors. |
| T-01-03 | D | startup/readiness path | mitigate | Fail fast on initial DB failure, expose liveness/readiness probes, and gate business routes when dependencies degrade. |
| T-01-04 | R | request tracing | accept | Optional trace IDs are accepted only from existing request IDs; no generated IDs in this phase per locked decision. |
</threat_model>
