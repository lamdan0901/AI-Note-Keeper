<threat_model>

## Trust Boundaries

| Boundary                              | Description                                                                    |
| ------------------------------------- | ------------------------------------------------------------------------------ |
| queue push payload -> push handler    | Replayed or malformed payloads can trigger duplicate sends or improper retries |
| push handler -> external FCM provider | External provider errors are untrusted and can create retry storms             |
| push handler -> token repository      | Cleanup writes must not delete unrelated user tokens                           |

## STRIDE Threat Register

| Threat ID | Category | Component                         | Disposition | Mitigation Plan                                                                                            |
| --------- | -------- | --------------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------- |
| T-05-10   | D        | transient retry loop              | mitigate    | Cap retries at parity policy (2) with 30s/60s delays and per-token scheduling (D-05, D-06)                 |
| T-05-11   | T        | token cleanup mutation            | mitigate    | Delete tokens only for explicit UNREGISTERED classification and scoped device/user identity (D-07)         |
| T-05-12   | R        | terminal failure observability    | mitigate    | Persist terminal failure records after retry exhaustion while continuing remaining token processing (D-08) |
| T-05-13   | E        | cross-token failure amplification | mitigate    | Isolate retries per token; never retry already-successful siblings (D-06)                                  |

</threat_model>
